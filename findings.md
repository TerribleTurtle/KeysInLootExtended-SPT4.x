# Architectural Discoveries (Disk Brain)

## Architecture & Initialization
*   **Initialization Hook:** The mod hooks into SPT using `IOnLoad` decorated with `[Injectable(TypePriority = OnLoadOrder.PostDBModLoader + 1)]`.
*   **Metadata Record:** Requires a manifest file using a record struct inheriting `AbstractModMetadata` (defines `ModGuid`, `Name`, `Version`, `SptVersion`).
*   **Dependency Injection:** Constructors in the hook class automatically receive injected services from SPT (e.g., `DatabaseServer`, `ISptLogger`, `FileSystem`).

## Configuration & Data Handling
*   **JSONC Parsing:** Configuration uses `.jsonc`. C# implementation MUST use `Newtonsoft.Json` (or `System.Text.Json` with `JsonCommentHandling.Skip`) to parse comments natively.
*   **JSON Caching (Performance):** Map config parsing loops through every map during startup. Allocating `JsonSerializerOptions` on every loop iteration spikes allocations. **[valid_from: 2026-07-15]** It must be globally cached in a `static readonly` field.
*   **String Normalization:** Profile switches (`config.ActiveProfile`) must always use `.ToLowerInvariant()` to prevent silent configuration failures if the user alters casing.
*   **Null-Coalescing Safety:** User-provided JSONC files may contain malformed or explicit `null` overrides (e.g. `"jacketContainer": null`). To prevent raw C# `NullReferenceException`s during object instantiation (where a serializer overrides `new()` defaults with `null`), use the null-coalescing operator `??` to gracefully fall back to global config weights.
*   **[2026-07-15] Switch Statements vs Dictionary Maps:** When scaling up large hardcoded configuration parameters (such as `switch (ActiveProfile)`), converting them into an O(1) `Dictionary<string, ProfileDefinition>` dramatically improves readability and maintenance. **Crucially**, `Dictionary.TryGetValue` throws `ArgumentNullException` on a null key (unlike a switch). Always protect dynamic user inputs using null-coalescing operators `TryGetValue(config.ActiveProfile ?? string.Empty, ...)`.
*   **Fail-Loudly Principles:** Misconfigurations or broken type checks must explicitly log and fail loudly instead of being swallowed.

## Core Loot Engine & Memory Optimizations
*   **Static Loot Manipulation:** Accessed via `DatabaseServer.GetTables().Locations[...].StaticLoot`.
*   **Dynamic Modded Maps:** Reflection (`db.Locations.GetType().GetProperties()`) must be used to dynamically fetch maps rather than relying on hardcoded lists, guaranteeing compatibility with custom user maps.
*   **Shared Array References (CRITICAL):** SPT heavily shares `ItemDistribution` array references across different maps to save memory. Mutating an array element in one map (e.g. Customs) will unintentionally affect other maps (e.g. Woods). When applying map-specific overrides (`locations/*.jsonc`), the mod must clone the reference (e.g. `Select(...)`) to ensure configurations remain isolated.
*   **Blast Radius (Int Overflow):** SPT uses standard 32-bit integers for relative probabilities. The sum of all container weights MUST be clamped to prevent exceeding `Int32.MaxValue` (`2,147,483,647`), which triggers an integer overflow and completely breaks map loot generation.
*   **The 0-Weight Trap:** If an injection engine evaluates a target weight as `0` and explicitly removes the item from the container dictionary (`distDict.Remove(id)`), it effectively purges that native item from the vanilla map pool. **[valid_from: 2026-07-15]** If the intent of `0` is "do not inject our custom weights, let vanilla take over," the script must `continue` without removing the existing vanilla object from the dictionary.
*   **Fractional Scaling & Minimum Bounds:** When applying fractional scalars (e.g. `0.05` for Loot Piñata), low-weight baselines can mathematically round down to `0`. **[valid_from: 2026-07-15]** Use `Math.Max(1, Math.Round(...))` to ensure existing low-weight keys aren't unintentionally purged from the ecosystem.
*   **Profile vs Custom Variables (Explicit Assignment):** **[valid_from: 2026-07-15]** Predefined profiles MUST explicitly hardcode and assign their own `ItemCountDistribution` arrays in C# rather than falling back to the raw JSON objects. If they fallback to the JSON objects, they inadvertently consume the user's "Custom Variables", violating the design intent that custom variables are only used when the profile is set to "Custom".
*   **Grid Resizing Limits (MaxCount Capping):** **[valid_from: 2026-07-15]** Expanding `CellsH` and `CellsV` on a container template is NOT enough to allow more loot to spawn. Item templates (like `5909e4b686f7747f5b744fa4` for Dead Scavs) have a hardcoded `MaxCount` property on their grid (vanilla is 5). The server's loot generator will silently discard all items generated beyond this `MaxCount`. When expanding a container grid, you MUST also raise `grid.Properties.MaxCount` (e.g. `CellsH * CellsV`) to physically allow the items to spawn.
## Mathematical Profiling & Mass Economy
*   **Profile Multipliers vs. Core Pools:** Key weights compete against the total mass of the *entire container* (e.g., 200,000 weight of junk vs. 300 weight of keys).
*   **Redistributing Probability Mass:** If a profile aims to reduce "trash" keys (e.g., multiplying Common weights by `0.3`), the total key mass in the container drops dramatically. This inadvertently lowers the overall frequency of finding *any* key. **[valid_from: 2026-07-15]** To achieve "less trash but the same amount of total keys", the lost probability mass must be aggressively offset by raising the Rare and SuperRare multipliers (e.g., `x1.7` and `x2.75`).
*   **NotExist Inverse Scaling (The Missing Keys Bug):** **[valid_from: 2026-07-16]** In SPT, the `Not_exist` RarityPvE string is applied to standard, common keys that have no specific rarity tag defined. It does **not** represent an empty container slot (empty slots are handled strictly by the `ItemCountDistribution` 0-count). `NotExist` must be properly scaled alongside other rarities (e.g., using `commonScale`); otherwise, standard keys will be completely excluded from map-specific generation.
*   **Simulated Supply & Demand (Economic Adjustments):** **[valid_from: 2026-07-16]** In Live Tarkov, massively increasing the key drop rate would naturally flood the Flea Market, crashing the sell price of keys. SPT's static flea market does not natively simulate this macro-economic crash. If a mod injects 5x more keys into the game without adjusting prices, the player exploits the static market and ruins progression. Profiles like "Balanced" and "Bountiful" MUST severely drop `KeyFleaPricesMultiplier` (e.g., to 0.4x or 0.2x) to mathematically simulate the market saturation caused by their respective drop rates. This also creates the psychological "Use It" Mentality, encouraging players to run rooms rather than treating keys as cash.
*   **Psychological Profile Fantasy (The Piñata Loophole):** **[valid_from: 2026-07-15]** While math dictates that extremely common items should be worthless, "cheat" profiles like "The Loot Piñata" rely on the psychological fantasy of getting rich quick. Applying a brutal economic crash (e.g., 5% prices) ruins the entire point of the profile. Treat extreme novelty profiles differently: give them the loot AND the vanilla 100% price tags.
*   **Vanilla Baseline Analysis:** **[valid_from: 2026-07-15]** Analysis of the native SPT `staticLoot.json` databases reveals that base game Jackets have a ~6% chance per item roll to spawn a key. However, this pool is highly restricted; high-tier keys are completely absent. Profiles like "Hardcore Scarcity" (5%) may statistically drop fewer keys than vanilla, but are vastly superior because they inject the *entire* key pool into the containers.

## Item Template & Database Manipulation
*   **Memory Optimizations (GC Pressure):** Looping over lists inside generic enumerations forces C# to allocate a closure Lambda per execution, heavily impacting GC during server boot. **Always convert lists to pre-computed `HashSet<T>` and execute O(1) memory-safe `Contains` operations.**
*   **[2026-07-15] Try-Catch Allocations inside Hot Loops:** Generating `new MongoId(...)` inside deep mapping loops alongside `try-catch` structures causes tremendous GC allocation spikes when strings fail parsing or when simply allocated en masse. Strings to be converted to `MongoId` must be parsed exactly *once* and stored outside of the core injection loops.
*   **Economy Adjustments:** Flea Prices accessed via `Templates.Prices`. Trader Assorts via `Traders[traderId].Assort`. Adjustments MUST use O(1) hash sets of injected keys to limit blast radius.
*   **[2026-07-15] Economy Pivot (Handbook over Assorts):** Instead of executing expensive O(N*M) iterations over every Trader's individual Assort list as originally planned, Trader Base Prices were pivoted to iterate over `tables.Templates.Handbook.Items` instead. The Handbook acts as the global base price index for all traders, making this a significant architectural optimization.
*   **MongoId Parsing Crashes:** Items added by other mods might contain invalid `MongoId` formats, causing server crashes when iterating arrays. The C# port must wrap `new MongoId(item.Id)` in a try-catch block for `FormatException` to skip malformed items and prevent server halts.
*   **Serialization & Testing:** Standard JSON deserialization fails on `MongoId` arrays. For isolated O(1) testing without heavy Server environments, mock properties via reflection using `System.Runtime.CompilerServices.RuntimeHelpers.GetUninitializedObject`.
*   **SPT Cache Hydration Race Conditions:** **[valid_from: 2026-07-15]** Internal SPT helper methods like `ItemHelper.IsOfBaseclass()` heavily rely on secondary caching services (e.g., `ItemBaseClassService`). Because C# mods load alphabetically, they can execute their `PostDBLoad` logic *before* the server fully builds these internal item hierarchy caches. If called too early, `IsOfBaseclass()` will silently return `false` for every item in the database. **Always perform manual, raw recursive traversal using `itemTemplate.Parent` from the live database dictionary instead of relying on SPT helpers for item categorization during initialization.**
*   **The "No-Duplicates" Static Loot Clamp:** **[valid_from: 2026-07-15]** The native `StaticLootGenerator` has a strict "no-duplicates" rule: it refuses to spawn multiple instances of the exact same static item in a single container. If a container's grid is set to spawn 25 items (`MaxCount = 25`, `ItemCountDistribution = 25`), but the container's loot pool only has 8 unique items, the generator will hit duplicates almost immediately and quietly abort, resulting in only 1-2 items physically spawning. You MUST aggressively populate the loot pool with massive amounts of unique items (e.g., all 242 game keys) for high-count distributions to function properly.
*   **SPT 4.0 LazyLoad Caching Removal (The "Ignored Loot" Bug):** **[valid_from: 2026-07-15]** In SPT 4.0, `LazyLoad<T>` was fundamentally refactored to remove result caching. Accessing `location.StaticLoot.Value` evaluates the underlying JSON file from disk every single time. If a mod directly mutates the dictionary returned by `.Value` in memory, it is mutating a temporary object that is instantly discarded. When the raid starts, the server requests `.Value` again and receives a pristine, 100% vanilla copy directly from the disk—bypassing the mod entirely. **Always use `.AddTransformer(dict => { ... return dict; })` to natively hook into the pipeline so the dictionary is modified on the fly during every read.**
*   **The Fence Simulation Loophole (Economy Leak):** **[valid_from: 2026-07-16]** In SPT, Fence's inventory is dynamically populated by emulating PMCs and Scavs extracting and selling their findings. Crucially, the simulator derives these findings directly from the map's `StaticLoot` pools. If a mod blindly injects *all* keys into the global loot pools, it will inadvertently inject Developer Test Keys and un-lootable Quest Items into the game world. Simulated PMCs will "find" these items and sell them to Fence, completely breaking the economy. You MUST filter injected items using `Item.Properties.QuestItem` and `ItemFilterService.IsItemBlacklisted()` before adding them to `StaticLoot`.
## Rejected Approaches
*   **[2026-07-15] Mutating `location.StaticLoot.Value` directly:** I initially attempted to modify the static loot pool by assigning values directly to the dictionary returned by `location.StaticLoot?.Value`. **Why it failed:** SPT 4.0's `LazyLoad<T>` class does not cache its output. Calling `.Value` parses the JSON file fresh every single time. Direct mutations only affected a temporary dictionary that was instantly garbage collected, meaning the server entirely ignored our mod's injected keys when loading into a raid. I pivoted to `.AddTransformer()` to register a callback that mutates the dictionary on the fly.
*   **[2026-07-15] Relying on `ItemHelper.IsOfBaseclass()` during `PostDBLoad`:** We initially attempted to find all keys in the database by calling the SPT core method `_itemHelper.IsOfBaseclass(item.Id, "543be5e94bdc2df1348b4568")` during the mod's initialization. **Why it failed:** C# server mods load alphabetically, meaning `KeysInLootExtended` executed its initialization hook before the server had fully hydrated its internal `ItemBaseClassService` cache. As a result, `IsOfBaseclass()` returned `false` for every item in the game, causing the mod to inject exactly 0 keys. We pivoted to manual `Parent` tree traversal of the raw `Templates.Items` dictionary to guarantee cache-independent accuracy.
*   **[2026-07-15] Purging `distDict` on `0` Weight:** We attempted to handle "0 weight" items by actively calling `distDict.Remove(mongoId)` to clear them from the mod's injected pool. **Why it failed:** Because the source list iterated over *all* keys in the vanilla database, this logic explicitly purged the keys from the native game map, overriding vanilla behavior entirely. We pivoted to a simple `continue;` statement to allow vanilla values to pass through unmolested.
*   **[2026-07-15] Uniform Fractional Reduction ("Vanilla Less Trash"):** We initially applied a flat `0.3x` scalar to Common keys to reduce trash, leaving Rare keys at `1.0x`. **Why it failed:** Due to the total container mass, this simply dropped the overall chance of finding a key by ~40%, rather than maintaining the same frequency but better quality. We pivoted to a multi-tiered scaling system to redistribute the lost mass.

*   **GitHub Release CLI Integration:** **[valid_from: 2026-07-15]** The `release.ps1` script natively parses the project version directly from `KeysInLootExtended.csproj` (using XML parsing) and automates pushing the compiled `.zip` artifact directly to GitHub Releases via the `gh release` CLI.
